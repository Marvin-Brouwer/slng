# `ADR` Masked value lazy reveal via postMessage

[csp-mdn]: https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Content-Security-Policy

The response panel webview displays HTTP responses that may contain sensitive values (API keys, tokens, auth headers, etc.) sourced from the user's environment variables or secret stores.
The panel must show these values are present while keeping them hidden by default, with an explicit user action required to reveal them.

The naive approach — serializing the full response AST including secret values into the webview HTML — would embed all secrets into the renderer process heap and DOM at page load, regardless of whether the user ever inspects them.

## Decision

Strip secret values from the AST before serialization. Represent sensitive nodes with a `mask` (display string) and a `reference` (numeric index into a lookup held by the extension host). \
When the user clicks reveal, the webview sends a `postMessage` to the extension host with the reference; the host resolves it and sends back only that value.

### Drivers

#### Reduced passive exposure surface

Secrets embedded in the initial HTML are immediately present in:

- The renderer process JavaScript heap
- The DOM (visible in devtools Elements/Memory tabs)
- Any VS Code webview state that may be persisted or restored (`getState`/`setState`)
- Screenshots or bug reports capturing the panel

With the reference-only approach, no secret ever enters the webview until the user explicitly requests it.

#### Explicit, auditable reveal

Each reveal is a discrete, intentional message exchange. The extension host is the sole authority for resolving references; it can log, rate-limit, or gate reveals without touching the webview.

#### `reference` is opaque without the host

A numeric index alone has no value to a passive observer. The extension host holds the only mapping from reference to secret.

### Alternatives

#### Serialize secrets into webview HTML

The simplest approach: pass the full response AST (including unmasked values) through `JSON.stringify` into the `<http-response>` element's text content.

This is dismissed because all secrets would be present in the DOM and renderer heap from the moment the panel loads, regardless of masking state in the UI.

#### Encrypt secrets before embedding

Embed encrypted values and decrypt in the webview using a session key passed via a separate channel.

This is dismissed because the decryption key must also reach the webview, reducing the security gain to obscurity. It also adds significant complexity for little practical benefit over the postMessage approach.

## Outcome

The webview never holds a secret value unless the user has explicitly revealed it in the current session. Clearing the panel (e.g. loading a new response) discards any revealed values from the renderer without any additional cleanup step.

### Pros and cons

`pro` Secrets absent from initial HTML payload — no passive leak via DOM inspection or screenshots \
`pro` Extension host is the single source of truth for secret resolution \
`pro` Reveal is a deliberate, traceable user action \
`pro` Strict [Content Security Policy][csp-mdn] on the webview prevents unauthorized script execution \
`pro` The `reference` integers are meaningless without the host-side lookup table

`con` Active [XSS][owasp-xss] in the webview could still enumerate references and call reveal for all of them \
`con` Reveal adds a round-trip latency (negligible in practice, host is in-process) \
`con` `event.source` origin validation is unreliable in VS Code webviews (host messages may arrive via an internal relay with `null` source); the CSP nonce on `script-src` is the effective XSS guard instead
